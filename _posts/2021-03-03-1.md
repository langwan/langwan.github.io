## 开发的摩尔定律

摩尔定律最早是英特尔创始人戈登·摩尔说的，集成电路可容纳的晶体管数量 18 个月就会增加一倍，作为姊妹的软件开发也是一样的，开发的功能也是越来越丰富。数据之间从简单关系到杂乱无章，当一个 APP 的功能达到一定数量，杂乱无章的数据调用就会让系统架构变的很难维护，不在适应接下来的变化速度。我们需要让数据的流动变的有序而简单。

## 从调用到订阅

我们以前从来不讲订阅，都是讲调用，你需要在界面上展示什么数据，你就调用，后端的 API 基本也就为你这个界面单独服务的，现在都讲订阅，API 和界面之间不在是简单的调用关系，而更多的开始考虑发布和订阅，多个界面订阅同一份数据源，在不同的地方展示商品的名称、价格和购买者的信息。

理想状态下，一个界面只需要订阅感兴趣的数据源就可以了，如果需要更新数据，就把数据推送出去就可以了。我们今天要讲的问题不出在订阅这个事情上，就来自推送。

## 倒置的界面树

手机 APP、桌面或是 WEB 都是大 VIEW 套着小 VIEW，所有的界面其实就是一棵树，一棵倒过来的树，从树根分叉到树冠，每个树冠就是我们看到的一个界面，树冠也是一棵更小的树。

数据就像一桶水，从上面浇下去，顺着树根，经过树冠一直流向树梢。我们希望树变成蓝色的，我们需要在树根上滴下一些派克笔的墨汁，蓝色会从树根渲染到树梢。这就是单向数据流动的最理想状态，虽然界面很复杂，但是数据都是被强制的从树根往下推送。这个更新过程在计算机的世界里，非常快，就像没有过程一样，例如我们改变了订单的状态，我们把这个状态的改变从树根这个地方推下去，那么整个界面就会顺从这个变化，刷出最新的界面。

如果我们希望登录界面是红色的，我们只需要在一个合适的树冠根部，可以是登录界面的根，也可以是登录界面这个路径上的任何一个根滴下红色的墨汁，那么这个根部下面所有的树冠都被染成了红色，而树的其它部分依然是那透明清澈的水。

所以即便是在同一个界面上我们需要变更一个信息，标题也好、状态也好、你需要思考清楚，你的信息要推送到那个根上，让这个数据像水一样再次流回来，看上去界面上就像是被直接更新掉了，实际上是重新推送上来的。

在这之前，数据的调用是杂乱的，当前功能怎么省事怎么来。现在我们并不需要思考那么多，只需要熟记这颗树，把数据变化推送到合适的位置就可以了。

## 自动更新的 UI

我用 React+Redux 最多，这一类界面库现在都是根据传来的数据自动刷新界面，所以你需要考虑的仅仅是如何规划这些数据，当需要更新的时候，把数据从合适的位置推送上来，这是趋势吧。

## 分离数据和界面状态

一共有三个关键的树根，数据往往要考虑推向这三个地方。

1. 树的根部，大部分的数据变更，例如直接调用 API 产生的结果，都可以直接推向这里，简单而有效，全部树就像烽火台，哪怕你是最边远的树梢也可以被通知到，而且时间延迟几乎无法被肉眼察觉。

2. 来自界面的展示需要，例如打开关闭侧边栏，图片被选中的状态，这些只存在于 APP 界面上的状态和信息，你推到树根就比较无聊了，你可以直接推到界面数据的根部，或者就是另外一棵树根，让 API 发生的结果可以不受影响的向上更新，界面的状态也不受影响的向下传递，互不干扰。例如这样一个极端的例子，一个用户选中了列表中的三张图片，但是其它用户删除了 1 张图片，导致这个共享界面被刷新了，用户看到的现象是图片少了一张，但是他选中的图片还处于选中状态。

3. 如果这个数据或者界面状态只是局部的，例如我们有一个图片的编辑器，图片编辑器里的图片的位置和状态仅仅和这个界面有关，你也不需要通知后台 API 什么，直接推送到它的根本就可以了，数据只会继续向下传播（流动）。

## 双向流动

VUE 通过 V-model 来实现双向数据流，简单来说就是 model 更新 View 就被更新，View 的数据更新也会反向的更新 model

有人说双向数据流是一个语法糖，确实，就是帮我们自动去推送了数据到一个绑定的数据上，数据的改变在单向的流动回来传播给周边的界面。

## 数据的版本

如果你的存款是 1 块钱，当你的存款变成 2 块的时候，这个数据明显被更新了，你会收到一个通知，你的存款是 2 块。但列表数据和一些复杂的对象就很难识别，如果前端是 NodeJS 你可以使用{...state}包裹数据，让 view 知道这个数据是新的，需要刷新界面，如果是其它语言，你可以在订阅数据的接口里，自动给数据打上一个时间戳。

如果一个列表的时间戳和之前订阅的数据发生了改变，那么界面就应该刷新了，例如一些桌面或者 APP 的开发，就可以统一利用这个方式，这就让所有语言都比较通用，这种方式比较简单，不需要多个层面去维护数据的版本，在订阅的口就可以做到了。
