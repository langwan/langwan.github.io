## 数据流动

早期程序数据从网络中发出去到界面上展现出来，并没有总结出数据流动这个概念，但这个概念被提出来以后，也非常容易理解。数据就像水一样从后端的起点不断分叉流向不同的界面，一旦数据被修改界面就会立刻被更新，我们管这样的能力叫做数据单向流动。

## 单向流程

前端组件里 React 我用的最多，它与 Redux 结合，完美的实现了数据的单向流动，当 Redux 里的数据被更新，订阅了数据 View 就会被重新渲染。

界面的嵌套关系是非常复杂的，一层又一层，所以 Redux 就像是数据的物流中心，后端发送来的数据被对齐在这里形成了新的数据层，大部分的数据变更（甚至应该是全部的）都应该在这里发生，任何层面的 View 都可以从这里订阅自己关心的数据让自己的界面被重新渲染。这就是我们故事里的小火车，大家都在等着他发货来更新自己。

你完全可以理解为后端的数据中心在 Database 上，而前端的数据中心就在 Redux 里面，而我们常说的数据单向流动都是指从 Redux 流向界面的这部分数据。

这就实现了一套界面里如果有多个地方展示了同样的内容，例如文章的标题，如果你变更了数据，只要新的数据被运往 Redux 物流中心，那么所有订阅这个数据的 View，无论你是嵌套了多少层以后的 View 都可以从这里直接拿走新数据让自己的重新渲染。

React 是一个很好的调用栈框架，它的下一层组件只承认上一层传递过来 Props，类似于函数的传参，传参被改变了组件才有机会被重新渲染。Redux 也不例外的把物流中心的数据包装成 Props 传递给组件，这样组件实际上就成了 Redux 的订阅者，一旦数据变化就会被重新渲染。（我们这里不关心组件的 state 这是组件的家务事，不是我们今天要讨论的目标，忽略它）

## 双向流程

VUE 通过 V-model 来实现双向数据流，简单来说就是 model 更新 View 就被更新，View 的数据更新也会反向的更新 model

有人说双向数据流是一个语法糖，我们今天要谈的就是这个问题。

## 实际上只有单向数据

这就像变魔术，你眼看的是 View 反过来把数据推回去了，但如果是我去实现，我会让数据去转圈圈，就像一个闭环的小火车，火车永远是顺时针方向移动，当老爷爷改变了状态，他就把数据推向数据中心，让数据顺着铁轨重新跑一圈，所以火车自始至终是顺时针方向移动的，但由于火车足够快，足够快，就给了人们一种错觉，好像有两列不同方向的火车，实际上只需要一列火车。

## 后端 RPC 推数据流

现在的技术已经不满足前端这么方便的开发模式，后端也想参与进来，彻底把任督二脉打通。

眼前最大的问题是多屏幕的问题，IOS、ANDROID、WEB、微信小程序、OSX、WINDOW 好多的屏幕，好多的技术，前端能干的技术都可以使用 React+Redux 来实现单向数据流，但目前而言其它语言的界面有不同的编程方，这还需要点时间。最终技术会书读同归，甚至变的一致。

对于前端而言，无论是 web 还是基于 web 开发的其它应用，道理上来说你不需要去破坏小火车幸福的小铁路，你只需要增加火车对后端数据的订阅通道即可，每次火车回到起始站去装货，后端给什么货就装什么，然后按照原来的逻辑跑圈就行了。

如果老爷爷想道歉，就让他打个电话 call 一下后端，让后端去更新数据，小火车回到起点，就会收到老爷爷的邀约，当跑到老奶奶这里的时候就收到道歉信了。所以理论上这并不难处理的样子。只是之前告诉小火车的话，兜了更大一圈告诉了后端。在程序的世界里，这和直接向老奶奶道歉没什么差别，太快了似乎就在当面道歉一样。

对于其它技术而言，没车的就赶快买车，没路的就赶快修路，让火车跑起来，就顺时针跑圈就可以了，数据流技术就统一了，当然你可以管自己叫优雅的单向数据流，也可以叫双向数据流，区别就是老爷爷是主动去找小火车说我要道歉（大量的显性的 onChange 事件）还是隐性的 V-Model，实际上也是触发了 OnChange，只是让你打字没那么累。就像 go 语言里的 if err == nil，那样遍地都是 onChange，没别的就是手累。

当然了既然小火车喜欢当这个邮差，后端主动推送数据变更才是主要的方式，你可以在 RPC 上去增加 PUB-SUB 属性就可以了，如果你不用 RPC，也可以把 PUB-SUB 定义在 REST 动作里面。

## 协议

协议上目前可以使用的就是 HTTP 2.0（GRPC 基于 HTTP2.0）和传统 WEB 的 socket.io，反正就是需要一个双向推送的长连接协议，可以用来订阅后端的数据，数据都在 REST 的 ACTION 里面，对于那些只读的列表或者数据，让小火车按照 URL(或者 RPC 里定义的路由，函数名什么的)远程监听着就行了。

## 游戏里的帧

有个朋友和我提到了这个事，按道理后端数据来的时候就可以发一列车，有很多顺时针跑着的小火车，如果你能把小火车限制成一辆，以每秒 60 圈的速度在跑，那就是游戏里的帧同步，那样效率是很高的，很节省 View 重新渲染的次数，但首先你的界面需要超过每秒 60 次的数据轰炸（一次更新就炸一次），否则无所谓。
